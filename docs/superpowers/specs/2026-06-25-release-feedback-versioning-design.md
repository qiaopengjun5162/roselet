# Roselet 稳定化版本设计

## 背景

Roselet 已经开始有真实用户访问，生产环境暴露出的问题不再只是“开发 bug”，还包括部署配置、静态构建产物、跨域、路由 fallback、版本追踪缺失等发布工程问题。下一阶段目标不是继续快速堆功能，而是先建立一个可追踪、可灰度、可回滚的稳定化流程。

本设计覆盖三件事：

1. 管理员可以查看用户具体反馈。
2. Web 关于页显示真实版本信息和用户可读的更新说明。
3. 发布流程从“push 后自动影响用户”改成“测试、预览、灰度、生产发布”的显式流程。

## 目标

- 管理员能在后台看到最近用户反馈内容，而不只看到反馈数量。
- 用户能在关于页看到当前版本和最近更新说明。
- 开发者能从线上页面反查到 Git tag / commit / 构建时间。
- 生产发布必须有明确版本号、发布说明、验证清单和回滚方式。
- 后续功能开发默认不直接影响线上用户。

## 非目标

- 不做完整 CMS 或运营后台。
- 不做用户可见的详细 Git commit 列表。
- 不把管理员后台公开给普通用户。
- 不把 GitHub token 或私有仓库信息暴露到前端。
- 不在本轮引入复杂的多环境数据库复制。

## 方案概览

推荐采用“轻量发布纪律”：

- Git tag / GitHub Release 是版本号和更新说明的权威来源。
- CI 构建时把版本号、短 commit、构建时间注入前端。
- 后端 `/health` 返回后端版本与 commit，便于排查前后端版本是否一致。
- `/stats` 管理后台增加“最近反馈”模块，复用现有管理员白名单。
- 新增 `docs/RELEASE_PROCESS.md`，固定开发、预览、灰度、生产、回滚步骤。

这套方案足够轻，不会引入大型发布系统；但关键链路都有记录，适合 Roselet 现在从 Beta 走向稳定试用。

## 管理员反馈收件箱

### 数据来源

现有 `feedbacks` 表已经保存：

- `id`
- `user_id`
- `content`
- `created_at`

因此本轮不需要新增数据库表。查询时左连接 `users`，展示用户昵称；匿名反馈或用户已删除时显示“匿名用户”。

### 后端接口

新增管理员接口：

```text
GET /api/admin/feedback?page=1&per_page=20
```

权限规则：

- 必须登录。
- 登录用户必须在 `ADMIN_USER_IDS` 内。
- 非登录返回 `401`。
- 已登录但非管理员返回 `403`。

返回结构：

```json
{
  "data": [
    {
      "id": 1,
      "user_id": "838fb4a6-25ba-4f1c-a7f6-96203e4741ad",
      "nickname": "paxon-admin",
      "content": "这里是用户反馈",
      "created_at": "2026-06-25T10:00:00Z"
    }
  ],
  "page": 1,
  "per_page": 20,
  "total": 1,
  "total_pages": 1
}
```

边界：

- 默认每页 20 条。
- 最大每页 50 条，避免后台一次拉太多。
- 按 `created_at DESC, id DESC` 排序。

### 前端展示

第一阶段直接放在 `/stats` 页面底部，标题为“最近反馈”。

展示内容：

- 反馈正文。
- 提交时间。
- 用户昵称或“匿名用户”。

失败状态：

- 反馈列表加载失败时，不影响统计卡片展示。
- 非管理员仍沿用当前 `/stats` 无权限页面。

### 测试

后端测试：

- 未登录访问反馈列表返回 `401`。
- 普通用户访问返回 `403`。
- 管理员访问返回反馈内容。
- 匿名反馈能显示 `nickname = null`。
- 分页参数受控。

前端测试：

- `/stats` 成功渲染反馈内容。
- 反馈列表加载失败时显示温和提示，不让整个后台失败。
- 非管理员仍显示无权限。

## 版本号与更新说明

### 版本来源

版本号以 Git tag / GitHub Release 为准，例如：

```text
v0.2.0
v0.2.1
v0.3.0
```

版本语义：

- `PATCH`：线上 bugfix、小文案、小兼容修复。
- `MINOR`：用户可感知的新能力或后台能力。
- `MAJOR`：破坏性变化，当前阶段尽量避免。

### 构建注入

前端构建时注入：

```text
NEXT_PUBLIC_APP_VERSION
NEXT_PUBLIC_COMMIT_SHA
NEXT_PUBLIC_BUILD_TIME
NEXT_PUBLIC_RELEASE_NOTES_URL
```

本地开发没有 tag 时允许显示：

```text
dev
```

生产构建不允许版本号为空；CI 需要在发布构建前检查。

### 关于页展示

`/about` 页面增加“当前版本”区域：

```text
当前版本：Roselet v0.2.0
构建时间：2026-06-25 18:30
提交：71ad693
更新说明：查看本次更新
```

给普通用户看的更新说明应来自 GitHub Release，而不是原始 commit message。commit message 偏工程化，Release 说明需要转换成自然语言。

示例：

```text
这次主要修复：
- 新增更容易打开的访问入口。
- 修复部分玫瑰详情打不开的问题。
- 优化声音开关体验。
- 管理后台可以查看用户反馈。
```

### 后端版本

`/health` 当前已有 `version` 字段，后续补充或确认：

```json
{
  "status": "ok",
  "database": "ok",
  "version": "v0.2.0",
  "commit": "71ad693"
}
```

如果前后端版本不一致，排查时能快速判断是前端部署滞后还是后端部署滞后。

## 发布流程

### 分支与环境

建议采用三层：

- `feature/*` 或 `codex/*`：日常开发，不能直接代表可发布版本。
- `main`：集成分支，必须通过 CI 和预览验证。
- Git tag / GitHub Release：生产发布入口。

生产部署原则：

- 后端生产部署从明确 tag 或手动 workflow 触发。
- Vercel / Cloudflare 生产部署应绑定稳定分支或 release 触发，不应让所有日常提交直接影响生产用户。
- PR / 分支预览用于验证页面和静态导出。

### 发布步骤

每个生产版本按以下顺序：

1. 开发分支完成实现。
2. 本地运行对应测试和构建。
3. 合并到 `main` 后等待 CI 通过。
4. 在 Vercel / Cloudflare 预览环境做人工冒烟。
5. 创建 release tag，例如 `v0.2.0`。
6. 编写 GitHub Release 用户向更新说明。
7. 触发生产部署。
8. 对两个线上入口做冒烟验证。
9. 在 `DEVLOG.md` 记录发布结果、问题和回滚点。

### 冒烟清单

生产发布后至少验证：

- 首页能打开。
- 登录/注册正常。
- 花圃能加载。
- 种一朵玫瑰成功。
- 新玫瑰详情页能打开。
- “听这朵玫瑰”不会卡在加载中。
- 关于页能提交反馈。
- 管理员 `/stats` 能打开并看到反馈列表。
- 普通用户不能访问 `/stats`。
- Vercel 和 Cloudflare 两个入口都通过核心路径。

### 灰度策略

当前阶段先采用“人工灰度”：

- 先在 Vercel 主入口验证。
- 再在 Cloudflare 入口验证。
- 确认两边核心路径正常后，再向群里发更新说明。

如果后续用户变多，再考虑更正式的流量切分或 feature flag。

### 回滚策略

前端：

- Vercel / Cloudflare 回滚到上一个成功部署。
- 如果是版本内容问题，重新发布上一个 tag 对应构建。

后端：

- Lightsail 使用 `~/roselet/.current_backend_image` 记录当前镜像。
- 发布前记录上一个 GHCR image。
- 失败时用上一镜像执行 `docker compose up -d --force-recreate backend`。

数据库：

- 本阶段避免不可逆迁移。
- 如必须做迁移，发布说明里单独列出“是否可回滚”。

## 实施顺序

建议拆成三个小 PR / commit 段：

1. `admin-feedback-inbox`
   - 后端管理员反馈列表接口。
   - Web `/stats` 最近反馈模块。
   - 单元/集成测试。

2. `version-visible-about-page`
   - 构建时注入版本信息。
   - 关于页展示版本和更新说明入口。
   - 后端 health 补充 commit。

3. `release-process`
   - 新增 `docs/RELEASE_PROCESS.md`。
   - 调整或记录 Vercel / Cloudflare / backend 的生产发布触发规则。
   - 增加发布冒烟清单。

## 验收标准

- 管理员登录 `/stats` 能看到最近反馈内容。
- 普通用户仍不能进入 `/stats`。
- 关于页显示版本号、构建时间、commit 和更新说明入口。
- `docs/RELEASE_PROCESS.md` 能指导一次完整生产发布和回滚。
- CI 和本地相关测试通过。
- `DEVLOG.md` 记录每次改动的根因、处理和验证。
