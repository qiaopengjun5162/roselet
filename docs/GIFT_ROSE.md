# 「送玫瑰」功能需求

> 把一朵玫瑰送给你想感谢的人。

---

## 动机

现在的 Roselet 是「为自己种花」。但如果我想感谢社区里的某个人呢？

「送玫瑰」让用户可以把一朵玫瑰送给指定的接收人——一种有仪式感的感谢方式。这和 PRODUCT.md 里的「情绪仪式」定位完全一致，而不是往社交网络方向走。

---

## 用户故事

1. 小明这周收到小红的帮助，决定种一朵红玫瑰送给她
2. 小明在种花时，除了填写感恩/焦虑/期待，还指定了接收人「小红」
3. 玫瑰出现在公共花圃中，但带有「小明→小红」的赠送标识
4. 小红在「我的花圃」里看到这朵别人送给她的玫瑰
5. 小红可以点赞回应（不需要回复文字）

---

## 交互流程

### 种花页

在现有的三种情绪输入框下方，新增一个可选字段：

```
┌────────────────────────────────┐
│  ... 现有的情绪输入框 ...       │
├────────────────────────────────┤
│  送给：（可选）                  │
│  ┌──────────────────────────┐  │
│  │ 输入对方的昵称...         │  │
│  └──────────────────────────┘  │
│  留空 = 为自己种花              │
└────────────────────────────────┘
```

- **可选字段**，不填就是现在的「为自己种花」
- 填写昵称后，按钮文案从「种下玫瑰吧」变成「送出这朵玫瑰吧」
- 如果填的昵称不存在，种花成功后给对方创建一个「占位用户」（下次对方登录时自动关联）

### 玫瑰卡片（花圃/详情页）

种花人一侧：
- 显示为「你送给 小红 的玫瑰」+ 原有内容

接收人一侧：
- 显示为「小明送给你的玫瑰」+ 原有内容

第三方：
- 显示为「小明送给小红的一朵玫瑰」

### 我的花圃

增加一个 Tab：「收到的玫瑰」，显示所有送给自己的玫瑰。

## 数据模型

### 新增字段到 `roses` 表

```sql
ALTER TABLE roses ADD COLUMN recipient_nickname TEXT;  -- 接收人昵称（冗余，方便显示）
ALTER TABLE roses ADD COLUMN recipient_user_id UUID REFERENCES users(id);  -- 接收人用户ID（可选，对方注册后关联）
```

### 与 `is_private` 的关系

| is_private | 有 recipient | 可见范围 |
|------------|-------------|---------|
| false | 无 | 所有人（现状） |
| false | 有 | 所有人 + 送出标记 |
| true | 无 | 仅自己（现状） |
| true | 有 | 自己 + 接收人 |

**规则**：有 recipient 的玫瑰，接收人永远能看到（即使 `is_private=true`）。

## API 变更

### POST /api/rose（种花）

新增可选字段：

```json
{
  "color": "red",
  "gratitude": "感谢小红的帮助",
  "anxiety": null,
  "hope": null,
  "is_private": false,
  "recipient_nickname": "小红"
}
```

### GET /api/my/roses（我的花圃）

新增 query 参数 `?view=sent` / `?view=received`：

- `?view=sent`（默认，现状）：我种的花
- `?view=received`：别人送给我的花

### 玫瑰详情

响应新增字段：

```json
{
  "recipient_nickname": "小红",
  "recipient_user_id": "uuid-or-null",
  "is_gift": true
}
```

## 边界情况

| 情况 | 处理 |
|------|------|
| 送给的昵称不存在 | 创建占位用户，后续对方注册/登录时用同一个昵称自动关联 |
| 送给自己 | 前端提示「花是送人的哦，送给自己直接种花就好」 |
| 接收人改名了 | `recipient_nickname` 是种花时的快照，不改。类似快递单上的收件人名字 |
| 多次送给同一个人 | 允许。每朵花独立存在 |
| 接收人删除自己账号 | recipient_user_id 变 NULL，nickname 保留（花的历史记录不丢失） |

## 实现优先级

### 第一步：核心（数据库 + API + 种花 + 详情页）

- 数据库迁移：`008_add_recipient.sql`
- 后端：CreateRose 支持 recipient_nickname 字段
- 后端：GET /api/my/roses 支持 `?view=received`
- 前端种花页：接收人昵称输入框（可选）
- 前端详情页：显示赠送关系

### 第二步：体验增强

- 前端花圃卡片：赠送标识
- 前端我的花圃：收到/送出 Tab 切换
- 种花按钮文案变化（送玫瑰 vs 种玫瑰）

### 第三步：通知（可选）

- 接收人下次登录时看到「有人送了你一朵玫瑰」提示
- WebSocket 实时推送
